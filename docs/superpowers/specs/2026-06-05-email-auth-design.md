# Email-вход через OTP-код (равноправно с Telegram)

**Дата:** 2026-06-05
**Статус:** дизайн утверждён, ожидает план

## Контекст и мотивация

Сейчас на `cv-builder.ru` единственный способ входа — Telegram (`@cvbuilder_support_bot`, deep link). Email-magic-link убрали 2026-05-28 (тогда платящих email-юзеров не было, а PKCE тащил кросс-браузерную боль). См. [[project_telegram_auth]].

Telegram-канал подвержен сетевым блокировкам РКН (см. историю с поллером/пином IP), поэтому логин, завязанный только на Telegram, — единая точка отказа. Добавляем **равноправный** email-вход, не зависящий от Telegram.

## Решения (из брейншторма)

1. **Роль:** email — равноправный вариант, всегда показан на `/auth` рядом с Telegram.
2. **Механика:** **OTP-код из письма** (6 знаков), ввод в той же вкладке. Полностью снимает кросс-браузерную боль (нет редиректов, нет PKCE).
3. **Отправка писем:** тот же email-пайплайн Supabase, что питал старые magic link (тот же SMTP, другой шаблон — с `{{ .Token }}`).
4. **Модель аккаунтов:** email и Telegram — **отдельные аккаунты, без связывания** (v1). Реальные email и фейковые `tg-*@telegram.cv-builder.ru` не пересекаются → коллизий нет. Один человек, вошедший обоими способами, получит два аккаунта с раздельными балансами — осознанный YAGNI для v1.

## Архитектура

Подход **A — клиентский блок** (отклонены: серверные роуты как у TG — оверкилл; magic link — кросс-браузер).

Используется браузерный Supabase-клиент (`@supabase/ssr` `createBrowserClient` из [lib/supabase.ts](resume-ai/lib/supabase.ts)) напрямую — он пишет сессию в куки, которые читают middleware и серверные компоненты.

**Флоу:**
1. Юзер на `/auth` отмечает согласие (общий чекбокс), опц. промо-код, опц. маркетинг.
2. Вводит email → «Получить код»:
   `supabase.auth.signInWithOtp({ email, options: { data: { consent_privacy, consent_marketing, promo_code, full_name: '' } } })`
   `shouldCreateUser` по умолчанию `true`. `options.data` → `raw_user_meta_data`, применяется триггером **только при создании нового юзера**.
3. На почту приходит 6-значный код. Юзер вводит код → «Войти»:
   `supabase.auth.verifyOtp({ email, token, type: 'email' })`
4. Успех → сессия в куках → редирект на `/dashboard`.

**Почему миграция БД не нужна:** триггер `handle_new_user` (миграция 010) уже корректно обрабатывает email-юзеров:
- `v_telegram_id := NULLIF(raw_user_meta_data->>'telegram_id','')::bigint` → `NULL`;
- `v_auth_provider` дефолтит в `'email'`;
- профиль создаётся с 50 токенами (+ промо-бонус), `telegram_id = NULL`, `auth_provider = 'email'`.

## Компоненты

- **`components/auth/email-auth-block.tsx`** (новый, клиентский). Назначение: весь email-OTP UI и логика. Props: `consentPrivacy: boolean`, `consentMarketing: boolean`, `promoCode: string`. Внутренний стейт: `step: 'email' | 'code'`, `email`, `code`, `sending`, `error`, `resendCooldown`. Зависимости: `createBrowserSupabaseClient`, `reachGoal` (метрика).
- **`app/(auth)/auth/page.tsx`** (правка). Под `TelegramAuthBlock` добавить разделитель «или» и `<EmailAuthBlock promoCode … consentPrivacy … consentMarketing … />` — те же пропсы из общего стейта формы.
- **Метрика:** новая цель `auth_email` (по аналогии с `auth_telegram`), стреляет при успешном `verifyOtp`.
- **Не трогаем:** серверные роуты `/api/auth/telegram/*`, `/callback`, миграции, middleware.

## UI

Одна карточка `/auth`:
- Заголовок/согласие/промо/маркетинг — как сейчас (общие для обоих способов).
- `TelegramAuthBlock` (без изменений).
- Разделитель «или».
- `EmailAuthBlock`:
  - **Шаг email:** input email + кнопка «Получить код» (неактивна без согласия).
  - **Шаг code:** input кода (6 цифр) + «Войти» + «Отправить заново» с кулдауном ~60с.
- Имя не спрашиваем — `full_name` пустой, заполнится при создании резюме.

## Внешняя зависимость: отправка письма

Единственный риск вне кода. Перед релизом проверить в **Supabase Dashboard → Auth**:
1. **Email Templates → Magic Link**: шаблон должен содержать `{{ .Token }}` (код), иначе OTP-код в письме не появится.
2. **SMTP**: настроен ли custom SMTP. Встроенная почта Supabase — лимит ~2-4 письма/час, для прода рискованно. Если custom SMTP нет — подключить провайдера (Resend/Postmark/Yandex) + SPF/DKIM для `cv-builder.ru`. Это операционная задача, не блокирует код.

## Обработка ошибок

| Случай | Поведение |
|--------|-----------|
| Неверный/просроченный код | «Неверный или устаревший код, запросите новый» |
| Rate-limit от Supabase (`429`/`over_email_send_rate_limit`) | «Слишком часто, попробуйте через минуту» |
| Согласие не отмечено | Кнопка «Получить код» неактивна |
| Сетевая ошибка `signInWithOtp`/`verifyOtp` | «Не удалось отправить/проверить код, попробуйте ещё раз» |

## Тестирование

- **Юнит (vitest):** валидатор email; редьюсер шагов компонента (email→code, ошибки, кулдаун resend).
- **Ручной чек-лист после деплоя:**
  1. Новый email → код приходит → вход → профиль с 50 токенами (+ промо, если введён).
  2. Повторный вход существующим email (без повторного начисления).
  3. Неверный код → ошибка.
  4. «Отправить заново» с кулдауном.
  5. Кросс-браузер: код, запрошенный в одном браузере, вводится в любом — работает (в отличие от старого magic link).
  6. Telegram-вход не сломан.
